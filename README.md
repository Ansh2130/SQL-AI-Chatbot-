# SQL - AI College Management Chatbot

An AI-powered chatbot for a **College Management System**. Staff and administrators ask questions in plain English — the system translates them into PostgreSQL queries, executes them safely, and returns a clear natural-language answer. Powered by Groq's LLaMA 3.3 70B model.

---

## What It Does

- **Natural language to SQL** — Ask anything about students, courses, faculty, fees, attendance, grades, and departments; the LLM writes and runs the query.
- **Forecasting** — Detects questions about future fee collection or revenue trends and routes them through a dedicated prediction pipeline that calculates YoY growth and projects ahead.
- **Read-only safety** — Every generated query is validated before execution. Only `SELECT` and `WITH` statements are allowed; destructive keywords are hard-blocked.
- **Auto chart generation** — The Streamlit UI asks the LLM to pick the best chart type (bar, line, or pie) for the data returned, then renders it inline.
- **Streaming responses** — The assistant's reply streams token-by-token in the Streamlit chat UI.
- **Session memory** — The last few conversation turns are included in every LLM call so follow-up questions work naturally.
- **Schema caching** — The database schema is introspected once and saved to `schema_cache.json`. A `/refresh_tables` API endpoint rebuilds it on demand.
- **Dual interface** — A Streamlit chat UI (`app.py`) for end users and a FastAPI REST API (`main.py`) for programmatic access.

## Database — College Management System

The chatbot is built on a PostgreSQL database with the following tables:

| Table | Key Columns |
|---|---|
| `public.students` | `roll_no`, `name`, `dept_code`, `admission_year`, `semester`, `cgpa`, `city`, `state`, `status` (`Active` / `Graduated` / `Dropped`) |
| `public.departments` | `dept_code`, `name`, `head_of_dept`, `building`, `established_year`, `status` |
| `public.courses` | `course_code`, `name`, `dept_code`, `faculty_code`, `credits`, `course_type` (`Core` / `Elective` / `Lab`) |
| `public.faculty` | `faculty_code`, `name`, `designation`, `dept_code`, `salary`, `status` |
| `public.enrollments` | `roll_no`, `course_code`, `academic_year`, `semester`, `grade`, `attendance_pct`, `status` (`Enrolled` / `Completed` / `Dropped` / `Failed`) |
| `public.fees` | `id`, `roll_no`, `fee_type` (`Tuition` / `Hostel` / `Library` / `Lab` / `Exam`), `amount`, `due_date`, `status` (`Pending` / `Paid` / `Overdue` / `Waived`) |
| `public.fee_payments` | `roll_no`, `fee_id`, `amount`, `payment_date`, `payment_method` (`Cash` / `Card` / `UPI` / `Bank Transfer`) |

### Table Relationships (from `langagent.py`)

```
students     ──── dept_code   ────▶  departments
students     ──── roll_no     ────▶  enrollments ──── course_code ──▶ courses
courses      ──── faculty_code ───▶  faculty
fees         ──── roll_no     ────▶  students
fee_payments ──── roll_no / fee_id ▶ students / fees
```

## Setup & Installation

### Prerequisites

- Python 3.9+
- PostgreSQL
- A [Groq API key](https://console.groq.com/)

### 1. Install dependencies

```bash
pip install -r requirements.txt
```

### 2. Configure environment variables

Create a `.env` file in the project root:

```env
GROQ_API_KEY=your_groq_api_key

# Option A — individual settings
DB_HOST=localhost
DB_PORT=5432
DB_NAME=your_db_name
DB_USER=postgres
DB_PASSWORD=your_password

# Option B — full connection URL (overrides the above)
DB_URL=postgresql+psycopg2://postgres:password@localhost:5432/your_db_name
```

### 3. Run the Streamlit UI

```bash
streamlit run app.py
```

### 4. Run the FastAPI server (optional)

```bash
uvicorn main:app --reload
```

Interactive API docs will be available at `http://localhost:8000/docs`.

## How Forecasting Works

When a question contains words like `forecast`, `predict`, `next year`, `projection`, or references a future year, the system:

1. `tools.py` — `detect_forecast_question()` identifies the intent and routes to `run_prediction_tool()`.
2. `langagent.py` — Called with `is_forecast=True`. Generates SQL that pulls historical fee collection grouped by `academic_year`, aliased as `totalsales` for the prediction engine.
3. `prediction_engine.py` — Receives the SQL results and:
   - `detect_grouping_intent()` checks if the user wants per-department or total forecasting.
   - `compute_summary_values()` calculates YoY growth rates and estimates the remaining months of the current year.
   - Calls the Groq API to narrate the forecast in structured format with a disclaimer.
4. The final forecast text is returned to the UI or API.

## SQL Safety Rules (from `tools.py` — `validate_sql_query()`)

- Only `SELECT` and `WITH ... SELECT` queries are permitted.
- Hard-blocked keywords: `INSERT`, `UPDATE`, `DELETE`, `DROP`, `ALTER`, `TRUNCATE`, `MERGE`, `EXEC`, `CREATE`, `GRANT`, `REVOKE`.
- Queries without a `LIMIT` clause have one added automatically (capped at 500 rows).
- Query execution enforces a 20-second timeout.
- On DB errors related to missing tables or columns, the schema cache is automatically rebuilt and the query retried (up to 3 attempts).
